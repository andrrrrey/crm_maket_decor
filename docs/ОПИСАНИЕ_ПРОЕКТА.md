## 1. Что это за система и какую задачу решает

**Maket Decor CRM** — это внутренняя рабочая система для студии, которая занимается
декором мероприятий (свадьбы, корпоративы, дни рождения и т.д.).

Вся работа студии — от первого звонка клиента до завершённого проекта на площадке —
ведётся в одной программе. Это позволяет не терять заявки, видеть нагрузку на команду,
контролировать склад и понимать, сколько денег приносит каждый менеджер.

Система закрывает сразу несколько задач, которые обычно ведут в разных табличках и чатах:

- **Воронка продаж** — входящие заявки превращаются в встречи, потом в сметы, потом в договоры.
- **Производство** — задачи по подготовке к мероприятию, закупки, монтаж/демонтаж.
- **Склад** — учёт инвентаря (декор, конструкции), тканей и цветов с фотографиями и остатками.
- **Дизайн** — макеты мероприятий, их статусы согласования.
- **Финансы** — суммы договоров, предоплаты, счета, расходы.
- **Коммуникации** — общий чат команды, чаты внутри проектов, уведомления.
- **Аналитика** — отчёты по менеджерам и периодам, воронка в цифрах.

Главная идея: **у каждой роли в студии — свой экран и свои права**, но все работают
с одними и теми же данными в реальном времени.

## 2. Общая картина: из чего собран проект

Если описать систему одним абзацем: это **веб-приложение на Next.js**, которое хранит
данные в **PostgreSQL** (через ORM Prisma), использует **Redis** для чата и фоновых
задач, отдаёт интерфейс на **React + Tailwind**, а в продакшене запускается в
**Docker** за **Nginx**.

Схематично весь «обвес» выглядит так:

```
            Пользователь (браузер)
                     │
                  [ Nginx ]   ← принимает HTTPS, раздаёт трафик
                     │
        ┌────────────┴─────────────┐
        │      Next.js + Socket.io │   ← основное приложение (веб + API + чат)
        └───────┬───────────┬──────┘
                │           │
         [ PostgreSQL ]   [ Redis ]
          (все данные)   (чат, очереди)
                              │
                        [ Worker ]   ← отдельный процесс: письма, проверка почты
```

Ключевая мысль: **это не одно приложение, а связка из нескольких самостоятельных частей**,
каждая из которых отвечает за свою зону. Почему так — разбираем в разделе 4.

## 3. Технологии и почему выбрали именно их

| Слой | Технология | Почему именно она |
|------|-----------|-------------------|
| Фреймворк | **Next.js 14 (App Router)** | Один фреймворк и для интерфейса, и для серверного API. Не нужно держать два отдельных проекта (отдельно сайт, отдельно бэкенд) — всё в одном месте, на одном языке. |
| Язык | **TypeScript** | Типы ловят ошибки ещё до запуска. В CRM много сущностей (клиент, договор, проект…), и типы защищают от путаницы между ними. |
| База данных | **PostgreSQL** | Надёжная реляционная СУБД. Данные в CRM сильно связаны между собой (клиент → договор → проект), а реляционная база как раз для этого. |
| ORM | **Prisma 5** | Позволяет работать с базой через удобный типизированный код вместо «сырого» SQL. Схема базы описана в одном файле, миграции генерируются автоматически. |
| Аутентификация | **NextAuth 5 (JWT)** | Готовое проверенное решение для входа. Не пишем свою криптографию и управление сессиями руками. |
| UI | **Tailwind CSS + shadcn/ui (Radix)** | Tailwind — быстрая вёрстка прямо в разметке. Radix даёт доступные, готовые компоненты (диалоги, выпадающие списки), не нужно изобретать их с нуля. |
| Таблицы | **TanStack Table** | В CRM много больших таблиц с сортировкой и фильтрами — эта библиотека рассчитана именно на такое. |
| Формы | **React Hook Form + Zod** | Удобные формы + проверка данных. Причём схема Zod используется **и на фронте, и на сервере** — одни правила в двух местах. |
| Реальное время | **Socket.io 4** | Чат и уведомления должны прилетать мгновенно, без перезагрузки страницы. WebSocket это и обеспечивает. |
| Очереди / кэш | **Redis 7 + BullMQ** | Redis хранит общее состояние чата между серверами и работает как «очередь задач». BullMQ — удобная обёртка над очередями. |
| Почта | **Nodemailer + IMAP** | Отправка писем (SMTP) и чтение входящих заявок с почты (IMAP). |
| Графики | **Recharts** | Готовые диаграммы для раздела аналитики. |
| Контейнеры | **Docker + Docker Compose** | Вся инфраструктура (приложение, база, Redis, Nginx) поднимается одной командой и одинаково работает на любом сервере. |

**Главный принцип выбора:** брать зрелые, широко используемые инструменты, которые
закрывают типовые задачи «из коробки», чтобы команда писала бизнес-логику, а не
переизобретала базовые вещи (вход, формы, таблицы, очереди).

## 4. Архитектура процессов: три приложения, а не одно

Это один из самых важных моментов для понимания проекта. В системе **три отдельных
запускаемых процесса**, и так сделано намеренно.

### 4.1. Веб-сервер (`server.js`)

Обычно Next.js запускают «как есть». Но здесь используется **кастомный сервер**
(`server.js`) — и вот почему: к стандартному Next.js «прикручен» **Socket.io** для
чата в реальном времени. Чтобы веб-приложение и WebSocket-сервер работали на одном
порту и в одном процессе, нужно создать HTTP-сервер вручную и передать его и Next.js,
и Socket.io.

```js
// server.js (упрощённо)
const server = createServer(...);   // создаём HTTP-сервер сами
const io = new Server(server, ...); // подключаем к нему Socket.io
io.adapter(createAdapter(...));     // подключаем Redis, чтобы чат работал на нескольких серверах
global.io = io;                     // делаем io доступным внутри API-роутов
```

> **Почему `global.io`?** Чтобы обычные API-роуты Next.js могли отправлять сообщения
> в WebSocket (например, «появилось новое уведомление»). Это простой способ дать им
> доступ к серверу Socket.io.

### 4.2. Worker (`worker.js`)

Отдельный фоновый процесс, который **не отвечает на запросы пользователей**, а делает
«тихую» работу в фоне:

- **отправляет письма** (очередь `email`);
- **создаёт уведомления** (очередь `notification`);
- **раз в 5 минут проверяет почтовый ящик** по IMAP — нет ли новых заявок с сайта (очередь `mailCheck`).

> **Почему это отдельный процесс, а не часть веб-сервера?** Потому что отправка письма
> или проверка почты могут быть медленными. Если делать это прямо в момент запроса
> пользователя, он будет ждать. Вынося такие задачи в фон, мы держим интерфейс быстрым,
> а тяжёлую работу выполняем отдельно. Если worker упадёт — сайт продолжит работать.

Интересная деталь: worker **не обращается к базе напрямую** для создания уведомлений,
а вызывает внутренний API веб-приложения по HTTP (с секретным заголовком
`x-worker-secret`). Так сделано, чтобы не дублировать логику работы с Prisma в двух
процессах — единая точка записи уведомлений живёт в API.

### 4.3. Инфраструктурные сервисы

PostgreSQL, Redis и Nginx — это не код проекта, а готовые сервисы, которые
запускаются рядом (через Docker Compose). О них — в разделе про развёртывание.

## 5. Структура папок проекта

```
crm_maket_decor/
├── prisma/
│   ├── schema.prisma      # описание всей базы данных (главный «источник правды»)
│   └── seed.ts            # начальные данные (например, пользователи)
├── src/
│   ├── app/               # Next.js App Router: страницы + API
│   │   ├── (dashboard)/   # все защищённые страницы интерфейса
│   │   ├── api/           # серверные эндпоинты (REST API)
│   │   ├── login/         # страница входа
│   │   └── page.tsx       # корневая страница (редирект)
│   ├── components/        # переиспользуемые React-компоненты
│   │   ├── layout/        # каркас интерфейса (шапка, меню, оболочка)
│   │   ├── tables/        # таблицы (клиенты, договоры…)
│   │   ├── stats/         # графики и карточки аналитики
│   │   ├── chat/          # окно чата
│   │   ├── files/         # загрузка и список файлов
│   │   ├── calendar/      # календарь проектов
│   │   ├── filters/       # фильтры (например, по менеджеру)
│   │   └── shared/        # мелкие общие штуки (бейджи, диалоги)
│   ├── lib/               # вспомогательная логика (не UI)
│   │   ├── auth.ts        # настройка входа (NextAuth)
│   │   ├── permissions.ts # кто что может видеть и менять
│   │   ├── prisma.ts      # подключение к базе
│   │   ├── queue.ts       # очереди задач (BullMQ)
│   │   ├── logger.ts      # запись действий в историю
│   │   ├── upload.ts      # работа с файлами на диске
│   │   ├── socket.ts      # подключение к WebSocket на стороне браузера
│   │   └── constants.ts   # подписи статусов, цвета и т.п.
│   ├── hooks/             # React-хуки (useSocket, useNotifications…)
│   ├── types/             # общие TypeScript-типы
│   └── middleware.ts      # проверка авторизации на каждый запрос
├── server.js             # кастомный веб-сервер (Next.js + Socket.io)
├── worker.js             # фоновый процесс (письма, почта, уведомления)
├── docker-compose.yml    # описание всех сервисов для запуска
├── Dockerfile            # как собрать образ приложения
└── nginx/                # конфигурация reverse-proxy
```

**Логика разделения папок:**
- `app/` — то, что видит пользователь, и то, что отвечает на его запросы.
- `components/` — «кирпичики» интерфейса, которые используются на многих страницах.
- `lib/` — «мозги» без интерфейса: правила доступа, подключения, утилиты. Такой код
  легко переиспользовать и тестировать.
- Структура `lib/` повторяет принцип «одна ответственность — один файл»: вход отдельно,
  права отдельно, файлы отдельно. Это упрощает поиск нужного кода.

## 6. Как устроена база данных

Вся база описана в одном файле — `prisma/schema.prisma`. Это «единый источник правды»:
по нему Prisma генерирует и код, и миграции. Меняем схему в одном месте — и всё
остальное подстраивается.

### Основные группы таблиц

Схема логически поделена комментариями на блоки:

- **Пользователи и роли** — `User`, `UserSettings`. У пользователя есть роль
  (`DIRECTOR`, `MANAGER`, `PRODUCTION`, `DESIGNER`) и персональные настройки.
- **Воронка продаж** — `Client` (входящая заявка/клиент) и `EstimateFile` (файлы смет).
- **Договоры** — `Contract` и связанные с ним задачи, закупки, сообщения, изображения,
  файлы, макеты (`ContractTask`, `ContractPurchase`, `ContractMessage` и т.д.).
- **Проекты и календарь** — `Project` (то, что попадает в производство и на календарь),
  `ProjectTask`, `ProjectPurchase`, `CalendarEntry`.
- **Дизайнер** — `Mockup` (макет) и его изображения.
- **Склад** — три независимых направления: инвентарь (`InventoryItem`), ткани (`Fabric`),
  цветы (`Flower`), каждое со своими категориями и фотографиями.
- **Справочники** — `Staff` (персонал), `Contractor` (подрядчики).
- **Прочее** — `ManagerTask` (задачи менеджера), `Expense` (расходы), `MailEntry`
  (входящие письма), `Message` (общий чат), `HistoryEntry` (история действий),
  `Notification` (уведомления).

### Ключевые связи и почему они такие

Главная цепочка системы повторяет реальный бизнес-процесс:

```
Client  ──конвертация──▶  Contract  ──────▶  Project
(заявка)                  (договор)          (производство/календарь)
```

- У `Client` есть необязательная связь с `Contract` (`contract Contract?`), а у
  `Contract` — поле `sourceClientId`. Когда заявку «превращают» в договор, эти записи
  связываются. Так сохраняется история: видно, из какой заявки вырос договор.
- У `Contract` есть необязательная связь с `Project`. Не каждый договор сразу проект,
  но когда дело доходит до производства — появляется проект.
- Списки задач, закупок, сообщений и файлов вынесены в **отдельные таблицы** со связью
  «один ко многим» и правилом `onDelete: Cascade`. Это значит: удалили договор — вместе
  с ним автоматически удалились его задачи и файлы. Не остаётся «мусора» в базе.

### Полезные приёмы в схеме

- **`@default(autoincrement())` для номеров** (`number`, `contractNumber`) — у клиента
  есть технический `id` (длинная строка cuid), но людям нужен простой человекочитаемый
  номер, поэтому он отдельным полем.
- **`Json`-поля** (`openMonths`, `yearPlans`, `details` в истории) — туда складывают
  гибкие данные, под которые не хочется заводить отдельные таблицы.
- **`Decimal` для денег** — суммы хранятся в типе `Decimal`, а не в обычном числе с
  плавающей точкой, чтобы не было ошибок округления в финансах.
- **Перечисления (`enum`)** для статусов — статус клиента, статус макета и т.п. жёстко
  ограничены списком значений. Нельзя случайно записать «непонятный» статус.

## 7. Бизнес-логика: путь клиента по системе

Чтобы понять смысл данных, полезно проследить реальный сценарий:

1. **Появилась заявка.** Менеджер создаёт `Client` (или заявка прилетает с сайта на
   почту, и worker заносит её в `MailEntry`). Статус — «Встреча».
2. **Идёт работа по воронке.** Статус клиента двигается: Встреча → Обсуждение → Смета →
   Договор. На этапе сметы к клиенту прикрепляются файлы (`EstimateFile`).
3. **Заявка превращается в договор.** Через эндпоинт `clients/[id]/convert` создаётся
   `Contract`, связанный с исходным клиентом. Появляются финансовые поля: сумма,
   предоплата, счёт.
4. **Договор уходит в производство.** На основе договора создаётся `Project` — он
   попадает в календарь и к производству. Заводятся задачи (`ProjectTask`) и закупки
   (`ProjectPurchase`).
5. **Команда работает над проектом.** Внутри проекта есть чат (`ProjectMessage`),
   изображения, чек-листы задач. Производство отмечает выполнение.
6. **Проект завершается.** Ставится `isCompleted = true`. Данные остаются для аналитики.

Параллельно дизайнер ведёт макеты (`Mockup`), а склад отслеживает остатки инвентаря,
тканей и цветов, которые используются в проектах.

**Почему данные разнесены по `Client` / `Contract` / `Project`, а не в одной таблице?**
Потому что это три разных этапа жизни с разным набором полей и разными правами доступа.
Заявка — это про продажи, договор — про финансы, проект — про производство. Разделение
делает каждую сущность простой и понятной, а права — точными.

## 8. Роли и права доступа

В системе 4 роли, и для каждой — свой набор доступных разделов. Вся логика прав собрана
в одном файле `src/lib/permissions.ts` — это удобно: чтобы понять «кто что может»,
достаточно открыть один файл.

| Роль | Кто это | Что видит |
|------|---------|-----------|
| **DIRECTOR** | Руководитель | Всё. Полный доступ ко всем разделам и аналитике. |
| **MANAGER** | Менеджер по продажам | Свои клиенты, договоры, календарь, склад, почта. Раздел «Инфо» — только если выдан особый доступ. |
| **PRODUCTION** | Производство | Календарь, производство, склад. |
| **DESIGNER** | Дизайнер | Только свои макеты и сообщения. |

Права работают на нескольких уровнях:

- **`canAccess(role, section)`** — может ли роль вообще открыть раздел (для меню и страниц).
- **`canWrite(role, section)`** — может ли роль не только смотреть, но и менять данные.
- **`shouldFilterByManager(role)`** — ключевой приём: менеджер видит **только своих**
  клиентов и договоры. В API к запросу автоматически добавляется условие
  `where.managerId = user.id`. Руководитель же видит всех и может фильтровать по менеджеру.
- **`shouldFilterByOpenMonths(role)`** — производство видит данные только за «открытые»
  месяцы (которые настроил руководитель).

> **Почему права вынесены в отдельный модуль, а не разбросаны по коду?** Потому что
> права доступа — это то, что меняется и проверяется чаще всего. Когда вся логика в
> одном месте, её легко аудировать, тестировать и менять без риска что-то забыть.

Отдельная тонкость: флаг `hasInfoAccess` (доступ менеджера к разделу «Инфо») при каждой
проверке сессии **подтягивается заново из базы**. Это сделано специально, чтобы
руководитель мог выдать или забрать доступ, и изменение применилось **без необходимости
пользователю выходить и снова входить**.

## 9. Аутентификация: как работает вход

За вход отвечает **NextAuth 5** (`src/lib/auth.ts`). Выбран самый понятный способ —
**логин и пароль** (Credentials provider).

Как это работает по шагам:

1. Пользователь вводит логин и пароль.
2. Сервер ищет пользователя по логину и проверяет, что он активен (`isActive`).
3. Пароль сверяется через **bcrypt** — в базе хранится не сам пароль, а его хеш
   (`passwordHash`). Даже при утечке базы пароли не раскрываются.
4. При успехе создаётся **JWT-токен** (стратегия `session: jwt`), который живёт 30 дней.
   В токен кладут роль, логин, тему оформления и другие нужные поля.

> **Почему JWT, а не сессии в базе?** JWT не требует похода в базу при каждой проверке —
> токен самодостаточен. Это быстрее и проще масштабируется на несколько серверов.

Защита маршрутов сделана через **`src/middleware.ts`** — это код, который выполняется
**перед** каждой страницей:

- если пользователь не авторизован и лезет в защищённый раздел — его перекидывает на `/login`
  (с запоминанием, куда он хотел попасть);
- если авторизован и открывает `/login` — его перекидывают на дашборд.

API-роуты дополнительно проверяют сессию сами (`const session = await auth()`), потому
что middleware намеренно **не трогает** пути `/api` (см. `matcher` в конце middleware).
То есть защита двухуровневая: middleware охраняет страницы, а каждый API-роут — сам себя.

## 10. Как устроен типичный API-запрос

Все серверные эндпоинты лежат в `src/app/api/`. Они построены по одному предсказуемому
шаблону. Разберём на примере создания клиента (`api/clients/route.ts`):

```ts
export async function POST(req: NextRequest) {
  // 1. Проверяем, что пользователь вошёл
  const session = await auth();
  if (!session) return NextResponse.json({ error: "Unauthorized" }, { status: 401 });

  // 2. Проверяем права роли
  const user = session.user as any;
  if (user.role !== "DIRECTOR" && user.role !== "MANAGER") {
    return NextResponse.json({ error: "Forbidden" }, { status: 403 });
  }

  // 3. Проверяем корректность данных через Zod
  const parsed = createClientSchema.safeParse(await req.json());
  if (!parsed.success) {
    return NextResponse.json({ error: parsed.error.flatten() }, { status: 400 });
  }

  // 4. Пишем в базу через Prisma
  const client = await prisma.client.create({ data: { ...parsed.data, managerId: user.id } });

  // 5. Логируем действие в историю
  await logAction(user.id, Actions.CLIENT_CREATE, "client", client.id, {...});

  // 6. Возвращаем результат
  return NextResponse.json({ data: client }, { status: 201 });
}
```

Этот единый шаблон **«проверка входа → проверка прав → валидация → работа с базой →
логирование → ответ»** повторяется почти во всех роутах. Что это даёт:

- **Предсказуемость.** Открыв любой роут, разработчик сразу понимает структуру.
- **Безопасность по умолчанию.** Каждый роут сам проверяет и вход, и права — нельзя
  «забыть» защиту.
- **Валидация в одном стиле.** Zod-схема описывает, какие данные допустимы; если данные
  кривые — запрос отклоняется с понятной ошибкой ещё до обращения к базе.
- **Прозрачность.** Любое важное действие пишется в `HistoryEntry` через `logAction`,
  и руководитель потом видит, кто что сделал. Причём ошибка логирования намеренно
  **не ломает** основной запрос (см. `logger.ts`).

Ответы стандартизированы: успешные данные приходят в поле `data`, ошибки — в поле `error`.

## 11. Реальное время: чат и уведомления

Чат и уведомления работают через **Socket.io** — это технология «живого» соединения
между браузером и сервером, по которому сообщения летят мгновенно в обе стороны.

Как это устроено:

- На сервере (`server.js`) Socket.io поднят рядом с Next.js. При подключении каждый
  пользователь «заходит» в свою личную комнату `user:<id>` — туда ему шлют персональные
  уведомления.
- Есть общая комната `general` (общий чат команды) и комнаты по проектам
  (`project:<id>`) — чтобы сообщения проекта видели только те, кто открыл этот проект.
- На стороне браузера подключением управляет `src/lib/socket.ts` и хук `useSocket`.
  Соединение одно на всё приложение (паттерн «синглтон») — это экономит ресурсы.

> **Почему к Socket.io подключён Redis-адаптер?** Если в будущем запустить несколько
> копий сервера (для нагрузки), пользователи окажутся подключены к разным копиям. Без
> Redis сообщение, отправленное на одном сервере, не дошло бы до людей на другом. Redis
> работает как «общая шина», через которую серверы обмениваются сообщениями. При этом,
> если Redis недоступен, сервер не падает, а просто переходит в режим «один сервер» —
> об этом заботится `try/catch` в `server.js`.

## 12. Фоновые задачи: почта и письма

Тяжёлые и медленные операции вынесены в **очереди задач** на базе **BullMQ** (поверх
Redis). Очереди объявлены в `src/lib/queue.ts`, а обрабатываются в отдельном процессе
`worker.js`.

Три очереди:

1. **`email`** — отправка исходящих писем через SMTP (Nodemailer). API кладёт задачу
   «отправить письмо», worker её забирает и отправляет.
2. **`notification`** — создание уведомлений. Worker дергает внутренний API, чтобы
   запись уведомления создавалась в одном месте.
3. **`mailCheck`** — **повторяющаяся** задача: раз в 5 минут worker заходит на почтовый
   ящик по IMAP, читает непрочитанные письма (это заявки с сайта), отправляет их в API
   (`/api/mail`) и помечает прочитанными.

> **Почему через очереди, а не напрямую?** Очередь делает систему устойчивой. Если письмо
> не отправилось (упал SMTP) — задачу можно повторить, она не теряется. Пользователь не
> ждёт отправки письма — он сразу получает ответ, а письмо уходит «в фоне». И всю эту
> нагрузку несёт отдельный процесс, не мешая основному сайту.

## 13. Работа с файлами

Файлы (сметы, договоры, фото инвентаря, макеты) хранятся **на диске сервера**, а не в
базе. В базе лежит только путь к файлу и метаданные (имя, размер, тип). Логика — в
`src/lib/upload.ts`.

Важные решения:

- **Уникальные имена файлов.** При загрузке имя превращается в `<cuid>_<очищенное_имя>.ext`.
  Случайный префикс исключает конфликты, а «очистка» имени до ASCII-символов гарантирует,
  что ссылки на файл будут работать корректно (без проблем с кириллицей и пробелами в URL).
- **Раздача через API, а не напрямую.** Файлы отдаются через роут `/api/files/...`, а не
  лежат в открытой папке. Это позволяет проверять права доступа перед выдачей файла.
  (В middleware путь `uploads` намеренно исключён из проверок, а контроль идёт на уровне API.)
- **Папки по типам** (`estimates`, `contracts`, `mockups`, `inventory`…) и ограничение по
  размеру/типу файла (`MAX_FILE_SIZE`, `ALLOWED_FILE_TYPES` в `constants.ts`) — чтобы
  нельзя было залить, например, исполняемый файл вместо сметы.

> **Почему файлы на диске, а не в базе?** Хранить большие файлы в базе — плохая практика:
> база раздувается и замедляется. Диск (подключённый как Docker-volume) для этого
> подходит лучше и проще бэкапится отдельно.

## 14. Frontend: как собран интерфейс

Интерфейс построен на **React** внутри Next.js App Router. Все рабочие страницы лежат в
группе `app/(dashboard)/` — скобки означают, что это «группа маршрутов» с общим
оформлением (шапка + боковое меню), но без отдельного сегмента в адресе.

Принципы организации интерфейса:

- **Единый каркас.** `components/layout/` содержит оболочку (`DashboardShell`), шапку
  (`Header`) и меню (`Sidebar`). Меню само скрывает разделы, недоступные роли, опираясь
  на те же права из `permissions.ts`. То есть права работают и на сервере, и в интерфейсе
  из одного источника.
- **Переиспользуемые компоненты.** Таблицы строятся на общей `DataTable` (TanStack Table),
  а конкретные (`ClientsTable`, `ContractsTable`) лишь настраивают колонки. Меньше
  дублирования.
- **Формы с проверкой.** React Hook Form + Zod. Та же Zod-схема, что и на сервере,
  проверяет данные в форме — пользователь видит ошибку сразу, не дожидаясь ответа сервера.
- **Хуки для логики.** `useSocket` (соединение), `useNotifications` (уведомления),
  `usePermission` (проверка прав в компонентах), `useWallpaper` (оформление). Логика
  отделена от внешнего вида.
- **Темы и оформление.** Поддержка светлой/тёмной темы (`next-themes`) и личных «обоев» —
  настройки хранятся у пользователя в базе и применяются после входа.

## 15. Развёртывание и инфраструктура

Вся система упакована в **Docker** и запускается через **Docker Compose**
(`docker-compose.yml`). Одной командой поднимаются пять сервисов:

| Сервис | Роль |
|--------|------|
| **app** | Веб-приложение (Next.js + Socket.io), порт 3000 |
| **worker** | Фоновый процесс (та же сборка, но команда `node worker.js`) |
| **postgres** | База данных PostgreSQL 16 |
| **redis** | Redis 7 для чата и очередей |
| **nginx** | Reverse-proxy: принимает 80/443, терминирует HTTPS, проксирует на app |

Важные детали инфраструктуры:

- **app и worker используют один и тот же Docker-образ**, просто запускаются с разными
  командами. Это упрощает сборку и гарантирует, что у них одинаковый код.
- **Постоянные данные вынесены в volumes** — база (`./data/postgres`), Redis
  (`./data/redis`) и загруженные файлы (`./uploads`). Контейнер можно пересоздать, а
  данные останутся.
- **PostgreSQL и Redis открыты только на `127.0.0.1`** — снаружи к ним не подключиться,
  доступ только изнутри сервера. Это базовая мера безопасности.
- **Healthcheck для базы** — app дожидается, пока PostgreSQL реально готов принимать
  соединения, прежде чем стартовать.
- **Секреты — через переменные окружения** (`.env`): пароль базы, секрет NextAuth,
  доступы к почте. В коде их нет (см. `.env.example` как образец).

Подробная пошаговая инструкция по установке на чистый сервер Ubuntu есть в
`docs/deploy-vps-ubuntu.md`.

## 16. Итоговая логика принятых решений

Если свести всё к нескольким принципам, которыми руководствовались при разработке:

1. **Один язык и один фреймворк на всё.** TypeScript + Next.js и для интерфейса, и для
   API. Меньше контекстов — проще поддержка маленькой командой.

2. **Тяжёлое — в фон, быстрое — пользователю.** Письма и проверка почты вынесены в
   отдельный worker и очереди, чтобы интерфейс всегда оставался отзывчивым, а сбои в
   фоне не роняли сайт.

3. **Права в одном месте и из одного источника.** Логика доступа собрана в
   `permissions.ts` и применяется одинаково и на сервере, и в меню. Это снижает риск
   «дыр» в безопасности.

4. **Безопасность по умолчанию в каждом роуте.** Единый шаблон API (вход → права →
   валидация → база → лог) делает защиту обязательной, а не опциональной.

5. **Данные следуют за бизнес-процессом.** Разделение `Client → Contract → Project`
   повторяет реальные этапы работы студии, поэтому модель данных понятна и людям, и коду.

6. **Зрелые инструменты вместо самописных.** Prisma, NextAuth, BullMQ, Radix, TanStack —
   проверенные библиотеки, которые закрывают типовые задачи и экономят время на главном —
   бизнес-логике.

7. **Готовность к росту, но без переусложнения.** Redis-адаптер для Socket.io и очереди
   позволяют при необходимости масштабироваться на несколько серверов, но при этом
   система спокойно работает и в простой одно-серверной конфигурации.
